# 代码开发指南

该仓库 fork 自 https://github.com/Wei-Shaw/sub2api, 并且需要定期合并 upstream 更新.
因此在该文档记录了在该仓库开发的注意事项.

## 1 确定修改代码是业务增强补丁

允许的提交:

- 添加明确的业务功能
- 修复真实的业务 bug
- 影响权限、计费、数据导入、幂等性、状态流转、安全等真实业务行为
- 最新代码仍然保留或依赖该行为

不允许的提交:

- 临时调试更改
- 代码格式化更改
- 旧结构兼容性更改
- 已被当前代码覆盖的更改
- 已废弃入口、路由、端点或旧交互的兼容性更改
- 无明确业务价值的杂项更改

## 2 代码规范

### 2.1 对新逻辑优先使用特定文件

特定文件指的是文件第一行带有`sudoapi`标记的文件.

在可行的情况下,将新的业务逻辑代码放入该特定文件中,如果该文件不存在则新建,第一行使用`sudoapi: <description>`
进行标记,然后是一个空行,之后是是文件的其余部分.

同一个业务逻辑必须使用完全相同的 description,并以英文句号结尾,例如:

```text
sudoapi: Idempotent admin data import.
```

Go file:

```go
// sudoapi: Idempotent admin data import.

package service
```

Vue file:

```vue
<!-- sudoapi: Idempotent admin data import. -->

<template>
```

TypeScript file:

```ts
// sudoapi: Idempotent admin data import.

export function parseRows() {
}
```

SQL migration:

```sql
-- sudoapi: Account contributor review workflow.

ALTER TABLE accounts ADD COLUMN review_status text;
```

### 2.2 最小化地修改现有文件

现有文件指的是来自上有仓库,第一行没有`sudoapi`标记的文件.

最小化地改动现有文件,应仅包含最小的集成点,避免:

- 导入顺序的更改
- 代码格式化不一致的更改
- 与当前业务无关的重构
- 旧入口/旧端点的兼容性恢复

不要在现有文件中随机插入代码,遵守以下插入规则:

- 尽可能将代码插入到相关逻辑组的最后部分,例如 route 放 group 末尾.
- 如果必须在现有文件中新增 helper,把他们放到文件末尾.
- 结构体/接口新增字段/方法,把他们放到末尾; 测试 fixture、DTO、TS interface 也遵守同样规则.
- 对象字面量新增字段也尽量放在该对象末尾,与对应结构体/interface 顺序保持一致.
- 将现有文件的多处修改尽量集中,使 diff 更小更清晰,例如相关 request/handler 放一起.
- 对于 Golang,考虑在原有代码和新增代码中间留一行空行,以避免`gofmt`重新对齐原有代码
- 对于 Golang 常量,优先使用独立的、带 marker 的 `const` 块; 必要时与上游常量块留空行,避免 `gofmt` 重新对齐上游代码.
- 如果 diff 块是新增的,或者是较多行数的修改(不包括删除),则在该块的起始处带上`sudoapi`标记.
- 对现有文件新增逻辑时,优先聚合成连续的 diff 块.

### 2.3 代码生成

对于以下代码,不应该直接修改,而是使用代码生成,工作目录是`backend`:

- `backend/cmd/server/wire_gen.go`
   ```shell
   go generate ./cmd/server
   ```
- `backend/ent` 中绝大部分代码 (文件开头标注`// Code generated by ent, DO NOT EDIT.`)
   ```shell
   go generate ./ent
   ```
- `backend/go.sum`
   ```shell
   go mod tidy
   ```

解决冲突时,先解决源文件冲突,再重新生成生成文件.不要手工把业务逻辑写进生成文件.

### 2.4 测试

#### 添加受影响行为的回归测试,至少覆盖相关影响面:

- 后端服务业务语义.
- Handler/API 请求与响应行为.
- 前端纯工具逻辑.
- 边界行为如幂等性、重复输入、跳过行、中止

#### 验证测试,根据影响面运行测试.

后端示例:

```shell
go test ./internal/handler/admin -run "<FeatureTest>" -count=1
go test -tags=unit ./internal/service -run "<FeatureTest>" -count=1
```

如果测试文件有`//go:build unit`,必须带`-tags=unit`,并确认输出没有`[no tests to run]`.

前端示例:

```shell
pnpm run typecheck
pnpm run test:run <test-file>
```
